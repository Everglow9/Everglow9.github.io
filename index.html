java总结（一）

2020-05-19

1、jar&jdk&jvm

（1）jdk：java的核心，包括java运行环境JRE、javag工具和Java核心类库

（2）jre：java运行必须环境，包含jvm实现和java核心类库

（3）jvm：java实现跨平台的核心，能运行Java语言开发的程序。java的源文件被编译成可在虚拟机上执行的字节码文件（跨平台性原因）

2、基础数据类型

（1）整型：byte(1字节)、short(2字节)、int(4字节)、long(8字节)

（2）浮点型：float(4字节)、double(8字节)

（3）布尔型：boolean

（4）字符型：char(2)字节

（5）对于short s1=1，s1=s1+1来讲，在s1=s1+1时进行了类型提升，int型再赋值给short类型变量时会出现类型转换错误；对于short s1=1,s1+=1来说，因为‘+=’为java规定的运算符，编译时会做特殊处理，可以正常编译。（注：自动类型转换是容量小的类型转换为容量大的类型）

3、基本数据类型包装类

（1）自动装箱是编译器将基本数据类型和对应的包装类型之间的转换；自动拆箱将包装类型转换为基本数据类型。如int<—->Integer

4、常量池

（1）String str=new String(“abc”)，编译器在堆中至少创建一个对象，也可能是两个。因为用到new关键字会在堆内存中创建一个str的String对象，value=”abc”。如果这个字符串在字符产量池中不存在，则会在常量池中创建这个String对象”abc”。

（2）“ ”创建的字符串在常量池中；如果字符串存在常量池中，则仅在堆内存中拷贝一份；如果不在，则先在常量池中创建再在堆中创建，共两个对象。

（3）intern：调用时，如果池中已包含一个等于此String对象的字符串，则返回池中字符串；否则池中直接存储该字符串入堆中。

4、引用类型

类、接口、数组都是引用类型。
（1）四种引用类型：避免对象长期占用内存

  • 强引用： GC不回收
  • 软引用：GC时JVM内存不足时会回收
  • 弱引用：GC时立即回收
  • 虚引用：GC时可能随时回收

    5、“==”和equals区别

    如果两个引用类型变量使用“==”运算符，则比较的是地址。equals默认与“==”相同，也是比较引用（地址）；但许多类重写了equals方法，如String、Integer等，将它变为了值比较，所以一般equals比较值是否相等。

（1）值相同的对象必须有相同的hashCode:

  • object1.equals(object2)为true时，hashCode相同；
  • hashCode不同时，object.equals(object2)为false；
  • hashCode相同时，object.equals(object2)不一定为true;

(2)如果重写equals不重写hashCode会怎样？

  • 向一个hash结构的集合中添加元素时，集合会首先调用hashCode方法，判断元素的存储位置。若该位置有其他元素时调用equals方法判断这两个元素是否相等。如果相等则不存，不相等则后移再判断。如果重写equals方法而不重写hashCode的话，两个值不同对象hashCode值一定不同，但执行重写后的equals结果为true；那么会导致HashSet或HashMap键会放入相同的对象。

    6、String&StringBuffer&StringBuilder

    都为final类，不允许继承；String长度不可变，其他两个可变；
    （1）StringBuffer是线程安全的，但效率低；StringBuilder是线程不安全的，但效率高。

Redis入门

2020-05-11

一、Redis

1、概念

redis是高性能的NoSQL系列的非关系型数据库

（1）、什么是NoSql(not only sql)：不仅仅是SQL，泛指非关系型数据库

2、NoSql和关系型数据库的比较

（1）、NoSql

①优点：

  • 开源、部署简单
  • 数据存储在缓存中，查询速度快
  • 数据以键值对的形式存储（key，value）,value支持多种数据格式；且不经过sql解析，性能高

②缺点：

  • 维护工具和参考资料有限
  • 不提供对sql致辞，学习成本高
  • 不提供关系型数据库对事务的处理

（2）、关系型数据库

①优点：

  • 可以复杂查询
  • 支持事务，使得安全性高

②缺点：

  • 查询效率相对较低

    3、命令操作

3.1：redis的数据结构

  • （1）、redis存储的是（key,value）格式的数据，其中key都是字符串，value有五种不同类型的数据结构
  • （2）、value的数据结构：字符串类型（String）、哈希类型hash（map格式）、列表类型list（linkedlis格式，支持重复元素）、集合类型set（不允许重复元素）、有序集合类型sortedset（不允许重复元素，且元素有序）

3.2：字符串类型String

（1）存储：set key value. 例：set username zhangsan

（2）获取：get key. 例：get username

（3）删除：del key. 例：del username

3.3：哈希类型hash

（1）存储：hset key field value. 例：hset myhash username lisi

（2）获取：hget key field. 例：hget myhash username

（3）删除：hdel key field. 例：hdel myhash username

3.4：列表类型list：可以添加一个元素到列表头部或尾部

（1）存储：

  • ①lpush key field value.（添加在列表左边） 例：lpush mylist a
  • ②rpushu key value.（添加在列表右边） 例：rpush mylist b

（2）获取：lrange key start end. 例：lrange mylist 0 -1（获取全部）

（3）删除：

  • ①lpop key.（删除列表最左边元素，并返回） 例：lpop mylist a
  • ②rpop key.（删除列表最右边元素，并返回） 例：rpop mylist b

3.5：集合类型Set：不允许重复元素

（1）存储：sadd key value. 例：sadd myset a

（2）获取：smembers key.（获取set集合中所有元素） 例：smembers myset

（3）删除：srem key value.（删除set集合中某个元素） 例：srem myset a

3.6：有序集合类型SortedSet：不允许重复元素,且有序。每个元素都会关联一个double类型的分数，redis通过分数来为集合成员进行从小到大的排序

（1）存储：zadd key score value. 例：sadd mysort 60 a

（2）获取：zrange key start end[withscores]. 例：zrange mysort 0 -1[withscore]

（3）删除：zrem key value. 例：zrem mysort a

3.7：通用命令

（1）key *：查询所有的键

（2）type key：获取键对应的value类型

（3）del key：删除指定的key value

4、持久化

4.1：redis是一个内存数据库，当redis服务器重启或者电脑重启时，数据会丢失。我们可以将redis内存中的数据持久化保存到硬盘的文件中

4.2：redis持久化机制

（1）RDB：默认方式，不需要配置，默认为该机制；在redis.windows.conf文件中有几项配置可以更改默认持久化的操作时间（在一定的时间间隔内，检测key的变化情况，然后持久化数据）

  • ①save 900 1 —– 15分钟进行了一次操作则存储
  • ②save 300 10 —- 5分钟进行了10次操作则存储
  • ③save 60 100000 —- 60秒进行了100000次操作则存储

（2）AOF：日志记录方式，可以记录每一条命令操作；编辑redis.windows.conf文件，将appendonly on改为appendonly yes，就开启aof日志记录方式

（3）删除：del key. 例：del username

  • ①#appendfsync always：每一次操作都进行持久化
  • ②#appendfsync everysec：每隔一秒进行一次持久化
  • ③#appendfsync no：不进行持久化

5、jedis中操作redis数据

注：使用redis缓存一些不经常发生改变的数据，数据库一旦发生改变，则需要更新缓存。数据库执行增删改时，需将redis缓存数据再次存入

Cookeie和Session

2020-05-06

一、会话技术

1、会话

一次会话中包含多次请求和响应；浏览器第一次给服务器资源发送请求，会话建立，直到一方断开为止。

2、功能

在一次会话范围内的多次请求访问间共享数据。

3、方式

（1）、客户端会话技术：Cookie

（2）、服务器端会话技术：Session

二、Cookie：一次请求访问多个资源

1、概念

客户端会话技术，将数据保存到客户端。

2、使用步骤

（1）、创建cookie对象，绑定数据：new Cookie(String name,String value)

（2）、发送collkie对：response.addCookie(Cookie cookie)

（3）、获取Cookie拿到数据：Cookie[] request.getCookies()

3、实现原理

基于响应头set-cookie和请求头cookie实现，存放键值对。

4、Coolie细节

（1）、一次请求能否发送多个cookie？

可以通过创建多个cookie对象，使用response多次调用addCookie。
（2）、Cookie在浏览器中的生命周期？

  • ①默认情况下，浏览器关闭后，cookie数据被销毁；

  • ②持久化存储:setMaxAge(int seconds)

    Ⅰ、正数：将cookie数据写到硬盘中，持久化存储，并指定cookie存活时间；时间到后cookie文件自动失效；

    Ⅱ、负数：默认值；

    Ⅲ、零：创建后立即删除cookie信息；

  • ③Cookie能不能存中文？

    在tomcat8之后支持中文，但不支持特殊字符

  • ④数据共享问题

    Ⅰ、在一个tomcat服务器中，部署了多个web项目，在这些项目中cookie能否共享？

    答：在默认情况下不能共享；在setPath(String path)中可以将path设置为“/”,默认为当前虚拟路径；

    Ⅱ、不同的tomcat服务器能否共享数据？

    答：默认不能，可以利用setDomain(String path)设置一级域名相同，那么多个服务器直接的cookie可以共享

    4、cookie的特点和作用

    （1）、特点

  • ①cookie存储数据在客户端浏览器；

  • ②浏览器对单个cookie的大小有限制（4kb），同一域名下的总cookie数量也有限制（20个）

（2）、作用

  • ①cookie一般用于存储少量的不敏感的数据，如配置信息；
  • ②在用户不登陆的情况下完成服务器对客户端的识别，因为cookie保存了某些用户的配置信息

三、Session

1、概念

服务器端会话技术；在一次会话的多次请求间共享数据，将数据保存在服务器端对象中（HttpSession）

2、使用步骤

（1）、获取HttpSession对象：HttpSession session = request.getSession();

（2）、使用HttpSession对象:

  • Object getAttribute(String name);
  • void setAttribute(String name,Object value);
  • void removeAttribute(String name);

    3、原理

    session的实现依赖于cookie的。利用cookie向客户端发送session的Id值并保存到客户端，其他session对象访问时会使用该id值，因此一个项目中的session对象只有一个。

    4、细节

    （1）当客户端被关闭后，服务器不关闭，两次获取Session是否为同一个

    答：默认不是；如果需要相同可以创建cookie，键为JSESSION,设置值为最大存活时间让cookie持久化保存

（2）客户端不关闭服务区关闭，两次获取的Session是否为同一个？

答：不是同一个；但要确保数据不丢失使用Tomcat自动完成

  • ①session钝化：在服务器正常关闭前将session对象序列号在硬盘上
  • ②session活化：在服务器启动后，将session文件转化为内存session对象即可

（3）session什么时候销毁？

  • ①服务器关闭，
  • ②sessi对象调用invalidate()，
  • ③sess默认失效时间到时(30分钟),

    5、Sessio特点

    （1）session用于存储一次会话中的多次请求数据，存在服务端

    （2）session可以存储任意类型、大小的数据

    6、Session和Cookie的区别

    （1）Session存储数据在服务器端；Cookie存储在客户端

    （2）Session存储数据大小没有限制；Cookie存储数据有大小限制

    （3）Session存储数据安全；Cookie相对不安全

2020-05-03

一、Response响应消息

1、概念

服务器端发送给客户端的数据

2、响应行

①组成：协议/版本、响应状态码、状态码描述（Http/1.1 200 ok）

②响应状态码：服务器告诉客户端浏览器本次的请求和响应状态。

  • Ⅰ、1xx：服务器收到客户端消息，但没有接收完成，等待一段时间后发送1xx状态码。
  • Ⅱ、2xx：成功200。
  • Ⅲ、3xx：重定向；302（重定向），304（访问缓存）
  • Ⅳ、4xx：客户端错误；404（请求路径没有对应资源）；405（请求方式没有对应的doxx方法）。
  • Ⅴ、5xx：服务器错误；500（服务器内部异常）

3、响应头

①格式：头名称：值

②常见响应头

  • Ⅰ、Content-Type：服务器告诉客户端本次响应体数据格式及编码方式；
    content-Type:text/html;charset=UTF-8
  • Ⅱ、Content-disposition:服务器告诉客户端以什么格式打开响应体数据（in-line：默认值，在当期页面打开；attachment;filename=xxx：以附件的形式打开响应体）。

③响应空行

④：响应体：传输的数据

二、Response对象

1、功能：设置响应消息

①：设置响应行：Http/1.1 200 ok

②：设置响应头：用于跳转重定向 setIteader(String name,String value)。
③: 设置响应体：

  • Ⅰ、获取字符输出流：PrintWriter getWriter();
  • Ⅱ、获取字节输出流：ServletOutputStream getOutputStream();

    2、重定向：资源跳转方式

    ①：功能
    • Ⅰ、设置转态码为302：response.setStatus(302);
    • Ⅱ、设置响应头为location：response.setHeader(“location”,”/test”);
    • Ⅲ、包含前两种方式：response.sendRedirect(“/test);

②：重定向特点：redirect

  • Ⅰ、地址栏发生改变
  • Ⅱ、重定向可以访问其他服务器资源
  • Ⅲ、重定向是两次请求，不能使用request对象来共享数据

③：路径写法：

  • Ⅰ、相对路径：通过相对路径不可以确定唯一资源，以.开头不以/开头；（如：./表示当前目录 ../后退一级目录）
  • Ⅱ、绝对路径：通过绝对路径可以确定唯一资源；以/开头（如：http://location/text/demon，/text/demon）。
  • Ⅲ、规则：通过判断路径给谁用以此来判断路径的写法；如果是给客户端浏览器使用，需要加虚拟目录（项目访问目录动态获取：request.getContextPath()）。如果给服务器使用不加虚拟目录。

服务器输出字符数数据到浏览器

  • Ⅰ、获取字符输出流
  • Ⅱ、输出数据
  • Ⅲ、乱码问题：response.setContentType(“text/html,charset:UTF-8”)

⑤：服务器输出字节数据到浏览器：

  • Ⅰ、ServletOutputStream
  • Ⅱ、getServletOutputStream

    三、ServletContext对象

    1、概念

    代表整个web应用，可以和程序的容器来通信；

    2、获取对象

    ①：通过request对象获取：request.getServletContext();

    ②：通过HttpServlet对象获取：this.getServletContext();

    3、功能

    ①：获取MIME类型：在互联网通信过程中定义的一种文件数据类型
    • Ⅰ、格式：大类型/小类型 text/html image/jpeg
    • Ⅱ、获取：String getMinmeType(String file)

②：域对象：共享数据

  • Ⅰ、setAttribute(String name,Object value)：设置数据
  • Ⅱ、getAttribute(String name)：获取数据
  • Ⅲ、removeAttribute(String name)：移除数据

③：获取文件的真实路径：String geRealPath(String path)

  • Ⅲ、serc目录下的资源访问：String c = context.getRealPath(“/WEB-INF/class/c.txt”)

Servlet入门

2020-04-29

一、Servlet

1、概念

servlet是运行在服务器端的程序。它是一个接口，其中定义了java类被浏览器访问的规则。

2、执行原理

①服务器收到客户端请求后，会解析URL路径，以此来获取访问Servlet的路径资源

②查找web.xml配置文件，查看相应的< url-pattern >标签体内容

③当查找第二步相应的标签后，再找对应的< servlet-class >中定义的全类名

④对应的服务器系统会将字节码文件加载进内存，并创建其相应的对象

3、Servlet的生命周期

①被创建时：执行init方法，并且只执行一次

  • servlet默认情况下在第一次访问时被创建；但可以配置servlet创建的时机。

    Ⅰ、当< load-on-startup >-1< /load-on-startup >中的值为负数，servlet第一次被访问时创建。

    Ⅱ、当< load-on-startup >1< /load-on-startup >中的值为正整数时，servlet在服务器启动时创建。
  • servlet的init方法只执行一次，说明一个servlet在内存中只存在一个对象；所以servlet是单例的，因此可能出现在多个用户访问时存在线程安全问题。为解决这个可能出现的问题在servlet中尽量不要定义成员变量；即使定义了也不要赋值。

②提供服务：执行service方式；每次访问servlet时都会调用一次service。

③被销毁：调用destory方法，只执行一次

  • servlet被销毁时执行，服务器关闭；servlet销毁之前执行destory。
  • 只有服务器正常关闭才会执行destory。
  • destory方法在servlet之前执行，一般用于释放资源。

二、Request

1、request对象和response对象的原理

①：request和response对象由服务器来创建。
②：request对象是用来获取消息，response对象是用来设置相应消息的。

2、request的功能

（1）获取请求行数据：GET/te/dem?name=zhang Http/1.1

  • ①：获取请求方式（GET)：String getMethod()
  • ②：获取虚拟目录（/te)：String getContextPath()
  • ③：获取Servlet路径（/dem)：String getQueryString()
  • ④：获取get方式请求参数（name=zhang)：String getQueryString()
  • ⑤：获取请求URL：

    Ⅰ、URI统一资源标识符（/te/dem)： String getRequestURI

    Ⅱ、URI统一资源定位符(http://localhost/te/dem):StringBuffer getRequestURL()
  • ⑥：获取协议版本（Http/1.1)：String getProtocol()
  • ⑦：获取客户机的Ip地址：String getRemoteAddr()

（2）获取请求头数据：

  • ①：通过请求头名称获取请求头值：String getHeader(String name)
  • ②：获取所有的请求头名称：Enumeration< String >getHeaderNames()

（3）获取请求体数据：只有post方式才有请求体

  • ①：获取字符输入流：BufferedReader getReader())
  • ②：获取字节输入流：ServletInputStream getInputStream()

    三、其他功能

    1、request对象和response对象的原理

    ①：request和response对象由服务器来创建。
    ②：request对象是用来获取消息，response对象是用来设置相应消息的。

    2、request的功能

    （1）获取请求参数通用方式：不论get还是post请求方式都可以使用下列方式获取请求参数
  • ①：根据参数名获取参数值：String getParameter(String name)
  • ②：根据参数名称获取参数值数组：String[] getParamterValues(String name)
  • ③：获取所有请求参数值数组：Enumeration< String > getParameterNames()
  • ④：获取所有参数的map集合：Map< String,String[] >getParamenterMap()

    注：中文乱码问题：tomcat8已解决get方式中文乱码问题;post方式先在获取参数前，设置request编码方式request.setCharacterEncoding(“UTF-8”).

（2）请求转发（forword）：一种在服务器内部的资源转发方式

①步骤:

IO流(1)

2020-04-22

一、字节流和字符流

1、概念

“流”代表能产生数据的数据源对象或可以接收数据的接收端对象；“流”屏蔽了I/O设备中处理数据的细节。java中IO类中流可以分为字节流和字符流；字节流基类为：InputStream和OutputStream;字符流基类为：Reader和Writer。

2、区别

字符流专门用于处理纯文本的数据；而字节流通常用于处于媒体文件等非纯文本文件；

3、如何选取合适的流对象

在编程过程中选取合适的流对象要做到三个明确。

1、明确源和目的：源(输入流)：InputStream、Reader

            目的(输出流)：OutputStream、Writer

2、明确操作数据是否为纯文本：是(字符流)：Writer、Reader

                     否(字节流)： OutputStream、Writer

3、明确具体的对象通过设备区分：源设备：内存、硬盘、键盘

                      目的设备： 内存、硬盘、控制台

二、输入流和输出流

1、概念

java中IO类分为输入和输出两部分。输入流（读取流）基类为InputStream和Reader，通常他们的派生类都含义read()方法；输出流（写入流）基类OutputStream和Writer,他们的派生类都含义writer()方法。

2、输出流(写入流)操作步骤

以FileWriter为例

  • ①：创建一个FileWriter对象，并指定被操作的文件。该文件会被创建到指定目录下，如果该目录下有同名文件则将覆盖；若未指定目录则会创建到当前对象的存储路径下。
  • ②：调用writer方法将字符串写入流中，并输出到指定文件中。
  • ③：刷新流对象中存储的数据，并将数据刷到指定的位置
  • ④：关闭流资源，关闭前会进行一次flush操作；和flush不同的是，flush刷新后可以继续使用该流对象；而close刷新后，流对象关闭。

    3、输入流(读取流)操作步骤

    步骤和输出流相差无几，但输入流中的close()将不会刷新流，而是直接关闭流。

三、读写/取缓存区

1、概念

缓冲区的出现是为了提高流的操作效率，在创建缓冲区之前，必须先有流对象

2、创建步骤

①：创建缓冲区之前写入流对象

③：只要到了缓冲区就要刷新

④：释放资源，关闭缓冲区就关闭了流对象

Java日记———Collection集合类(1)

2020-03-30

一、Collection容器

1、概念

Java有多种方式去保存数组，其中数组是保存一组对象的最有效的方式。但数组具有固定的大小，且存储对象的形式相对简单。在实际的编程过程中使用数组来存储对象非常不方便。因此Java类库提供了一套容器来解决在实际编程存储对象受限的问题。Java集合类使用Collection来指代该框架的所有类。其中基本的类型是List、Set、Map、Queue。

2、迭代器

集合中取出元素的形式；把取出方式定义在集合的内部，这样取出方式就可以直接访问访问集合内部的元素，取出方式定义为内部类。虽然每个容器内部的数据结构都不一样，但他们的内部都符合迭代器（Iterator）的规则，可以通过对外提供的方法iterator()来获取集合中的对象。

二、List集合

1、概念

List元素是有序的且元素是可重复的，集合体系中有索引，因此凡是可以操作角标的方法都是该体系的特有方法

2、List集合的三种类型

  • ArrayList: 底层使用的数据结构是数组结构；因此ArrayList具有数组的查询速度快的特点，但其增加和删除元素的速度相对较慢，且该集合类是线程同步的。

  • LinkedList:底层使用的数据结构是链表结构；具有增、删数据快，但查询数据相对较慢的特点，该集合类线程不同步。

  • Vector:底层使用的数据结构是数组结构，该集合线程同步，被ArrayList替代。

3、List集合特有的迭代器

ListIterator是Iterator的子接口；在迭代时不能通过集合对象的方法操作集合中的元素对象，否则会发生ConcurrentModificationException异常。所以在迭代时，只能用迭代器的方法操作元素对象；但Iterator方法有限，只能对元素进行判断、取出以及删除操作。如果想要进行其他操作如：添加、修改等，就需要使用其子接口ListIterator，该接口只能通过List集合的对象来获取。

三、Set集合

1、概念

Set中元素是无序的，存入和取出顺序不同。保存的元素不能重复，Set的功能与Collection一致，实际上Set就是Collection,只是行为不同

2、List集合的三种类型

  • HashSet: 底层数据结构是哈希表，是为了快速查找而设计的Set类型。线程不同步且存入元素必须定义hashCode()。

  • TreeSet:底层使用的数据结构是二叉树。可以实现对Set集合中的元素进行排序。元素必须实现Comparable接口，并利用compareTo方法来保证元素的唯一性。线程不同步

  • LinkedHashSet:具有HashSet的查询速度，且内部使用链表维护元素的顺序。元素必须定义hashCode()。

三、Stack栈和Queue队列

1、概念

（1）栈：“栈”通常指的是后进先出的容器。

（2）队列：队列是一个先进先出的容器。

JAVA日记——多线程（2）

2020-03-10

线程锁

在多线程运行可能会出现争夺资源的问题，这可以导致数据和程序出现重大的错误；为了防止这种冲突的方法就是当资源被另一个任务使用时，对该资源上锁。第一个访问某个资源的线程对其上锁，在线程任务结束后解锁，下一个访问该资源的线程继续对其上锁并使用该资源，以此类推。

JAVA中提供了两种上锁方式，分别是关键字synchronized和接口Lock。

一、synchronized关键字

1、概念： 当任务要执行被关键字synchronized关键字保护的代码时，将会检查锁是否可用，然后获取锁，执行代码然后释放锁,synchronized通过jvm实现锁机制，线程执行完自动释放锁。

2、synchronized锁的对象是谁

①: 如果synchronized修饰非静态方法，则锁定的是方法的调用者，也就是this对象。可以通过对象的个数看出线程是否同步；如果存在一个对象则同步，存在两个或两个以上则不同步。

②：如果synchronized修饰的是静态方法，则锁定的是类；此时无论创建多少个对象，线程都是同步的。

③：如果synchronized既修饰静态方法又修饰非静态方法，则静态锁类，非静态锁对象，互不干扰。

④：如果synchronized修饰的是代码块，则锁定的是传入对象(synchroinzed(num) ),如例所示，也就是num对象；能否同步就看锁定的对象是否为同一个对象。

二、Lock锁

Lock是对synchronized的升级，是一个JUC接口，常用的实现类是ReentrantLock(重入锁)。Lock对象必须被显示的创建、锁定和释放，对于内建锁来说相对复杂，但其灵活性更高。

三、Lock锁和synchronized

  • synchronized是关键字，Lock是接口
  • synchronized通过jvm实现锁，Lock通过jdk来实现
  • synchronized自动上锁、自动解锁，Lock手动上锁、手动解锁
  • synchronized无法判断是否可以获得锁，Lock可以判断是否拿到锁
  • synchronized拿不到锁会一直等待，Lock不一定会一直等待
  • synchronized是非公平锁，Lock可以设置是否为公平锁

四、死锁

某个任务在等待另一个任务，而后者又等待别的任务，一直下去，直到这条等待链上的任务在等待第一个任务释放资源。就形成了任务之间相互等待的连续循环，导致没有那个任务线程可以继续，就形成了死锁。

1、死锁产生的四个条件：

当这四个条件同时满足时，就会产生死锁

①: 互斥条件: 任务使用的资源中至少有一个是不能共享的

②：保持请求：至少有一个任务持有一个资源，并且等待获取另一个被别的任务持有的资源

③：不剥夺条件：资源不能被任务抢夺。

④：循环等待：一个任务等待另一个任务持有的资源，而后者又等待别的任务释放资源，一直这样下去

多线程

JVM启动时会有一个java.exe的进程，该进程中至少有一个线程负责java程序的执行，当线程运行的代码存储的main函数中时，称该线程为主线程；JVM启动时还会启动一个负责垃圾回收机制的线程。

1、进程：是指一个程序执行的过程，每个进程都有一个执行顺序，该顺序由一个控制单元控制。

2、线程：进程中的一个独立的控制单元，线程控制着进程的执行；一个进程包含一个或多个线程。

3、自定义线程方法

（1）方法一： 继承Thread类

(1)：实现方式可以避免单继承的局限性，类可以在实现进程的情况下继承其他自己所需要的类，增加了程序的扩展性
